# Завдання

## Загальна архітектура системи

Усі компоненти розгортаються на одній віртуальній машині Linux.

Архітектура системи:

client → nginx (reverse proxy) → web application → SQL database

Мережеві обмеження:

| **Компонент** | **Адреса** | **Порт**        |
|---------------|------------|-----------------|
| nginx         | 0.0.0.0    | 80              |
| web app       | 127.0.0.1  | Згідно варіанту |
| SQL БД        | 127.0.0.1  | Згідно варіанту |

Вимоги:

- Доступ до бази даних має бути обмежений лише з віртуальної машини

- клієнти взаємодіють із системою через reverse proxy (nginx)

## Реалізація веб-застосунку

Реалізувати веб-застосунок згідно варіанту індивідуального завдання. Для реалізації дозволяється використовувати будь-які мови програмування та фреймворки. Окрім реалізації бізнес-логіки, описаної в варіанті індивідуального завдання усі варіанти веб-застосунку мають реалізовувати:

- Ендпоінти для перевірки стану.

  - GET /health/alive --- завжди повертає HTTP 200 з вмістом OK

  - GET /health/ready --- повертає HTTP 200 з вмістом OK якщо сервіс успішно підключився до БД і готовий обробляти запити, інакше повертає HTTP 500 з коротким описом, чого сервісу не вистачає

<!-- -->

- API, яке реалізовує бізнес-логіку повинне віддавати різні результати залежно від заголовку [[Accept]{.underline}](https://developer.mozilla.org/en-US/docs/Web/HTTP/Reference/Headers/Accept). Якщо клієнт очікує text/html --- віддавати просту html-сторінку з вказаним вмістом (plain html, без js, без стилів, для списків виводити дані у таблиці), якщо клієнт очікує application/json --- то віддавати дані у json форматі..

- Кореневий ендпоінт, який [[очікує і віддає]{.underline}](https://developer.mozilla.org/ru/docs/Web/HTTP/Reference/Headers/Accept) тільки text/html і повертає список усіх ендпоінтів бізнес-логіки застосунку.

Для збереження даних веб-застосунок повинен використовувати базу даних згідно варіанту

## Скрипт міграції бази даних

Обов'язковою частиною розроблюваного веб-застосунку має бути скрипт міграції бази даних. Його задача підключитися до БД та створити усі необхідні для роботи сервісу таблиці та індекси. Скрипт передбачає, що надане йому підключення містить або порожню БД (без жодного об'єкту), або БД цієї ж або попередньої версії застосунку, створену за допомогою цього скрипта.

## Розгортання сервісу

Для виконання роботи необхідно розгорнути веб-застосунок, базу даних та reverse-proxy на віртуальній машині з Linux.

### Systemd для веб-застосунку

Ваш веб-застосунок має запускатися за допомогою [[systemd-unit]{.underline}](https://www.freedesktop.org/software/systemd/man/systemd.unit.html). Unit-файл для веб-застосунку має знаходитися за шляхом /etc/systemd/system/mywebapp.service. Сервіс повинен працювати від окремого користувача в системі mywebapp та виконувати [[міграцію бази даних]{.underline}](#скрипт-міграції-бази-даних) перед запуском.

Після реалізації простого systemd-unit переробити запуск на використання [[systemd socket activation]{.underline}](https://ilmanzo.github.io/post/systemd-socket-activated-services/).

### Reverse Proxy

Необхідно налаштувати Nginx як reverse proxy для застосунку. Nginx має слухати на порту 80 (стандартний HTTP порт), писати лог запитів до сервісу. Назовні nginx повинен віддавати лише кореневий ендпоінт та ендпоінти бізнес-логіки застосунку.

## Скрипт автоматичного встановлення

Автоматизувати розгортання віртуальної машини. Автоматизація має бути реалізована у вигляді одного-декількох скриптів (bash/python) та мати єдину точку входу. Автоматизація при запуску повинна:

1.  встановити необхідні пакети

2.  створити користувачів

3.  створити базу даних

4.  скопіювати або згенерувати конфігураційні файли

5.  встановити systemd-сервіс для веб-застосунку

6.  запустити сервіс

7.  налаштувати nginx

8.  (обов'язково) створити текстовий файл /home/student/gradebook який містить лише одне [[число N]{.underline}](#варіанти-індивідуальних-завдань), від якого ви розраховували варіант завдання.

Після виконання скрипта система повинна працювати. Також скрипт повинен блокувати "дефолтного" користувача в системі, щоб після первинної конфігурації можна було увійти тільки з-під створених користувачів (student, teacher, operator).

# Результат роботи

## Git репозиторій

Проєкт повинен бути оформлений у вигляді публічного Git-репозиторію. Репозиторій повинен містити:

- Реалізований веб-застосунок

- Автоматизацію для розгортання

- Документацію

Історія комітів повинна бути логічною та зрозумілою, відображати хід виконання роботи студентом.

## Документація

Репозиторій повинен містити документацію по виконаному завданню. Точкою входу в документацію має бути файл README.md або README.rst в корені репозиторію. За потреби можна створювати додаткові файли документації у директорії docs, та посилатися на них із файлу README.

Документація повинна містити:

- Варіант індивідуального завдання (розрахунки і опис конкретного індивідуального завдання)

- Документацію по розробленому веб-застосунку

  - Призначення застосунку

  - Як налаштувати середовище для розробки/тестування/запуску розробленого застосунку

  - Як запустити розроблений веб-застосунок

  - Документацію по API-ендпоінтах

- Документацію по розгортанню

  - Де взяти базовий образ віртуальної машини (використовуйте тільки офіційні образи дистрибутивів, наприклад, [[Ubuntu]{.underline}](https://ubuntu.com/download/server), [[CentOS]{.underline}](https://www.centos.org/download/), тощо). Який образ завантажувати

  - Вимоги до ресурсів віртуальної машини (CPU, RAM, Disk, тощо)

  - За потреби --- чи потрібні якісь спеціальні налаштування при встановленні OS на віртуальну машину (розбивка диску, тощо)

  - Як увійти на ВМ (SSH, Console) та які credentials використовувати (для користувача за замовчуванням)

  - Як завантажити/запустити вашу автоматизацію розгортання

- Інструкцію з тестування розгорнутої системи, а саме --- як ви перевіряли коректність того, що ви налаштували.
